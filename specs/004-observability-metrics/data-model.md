# Data Model: Observability for Service Operations

## Overview

This feature introduces request-scoped observability data and runtime configuration, but it does not add or modify persistent business entities. The data model is operational and transport-oriented rather than domain-oriented.

## Runtime Configuration

### ObservabilitySettings

- **Purpose**: Controls tracing and metrics behavior at service startup.
- **Fields**:
  - `log_level`
  - `request_id_header`
  - `metrics_enabled`
  - `metrics_path`
- **Validation/invariants**:
  - `log_level` must map to a supported tracing filter string or a safe default.
  - `request_id_header` must resolve to a valid HTTP header name.
  - `metrics_path` must be a valid route path when metrics are enabled.
  - Defaults must allow local startup without additional configuration.

## Request-Scoped Observability Data

### RequestCorrelation

- **Purpose**: Represents the effective correlation identifier for one inbound HTTP request.
- **Fields**:
  - `incoming_request_id: Option<String>`
  - `effective_request_id: String`
  - `response_header_name: String`
- **Rules**:
  - If an inbound request ID is present, `effective_request_id` preserves it.
  - If no inbound request ID is present, `effective_request_id` is generated by the service.
  - The same `effective_request_id` must be used in tracing context and the HTTP response.

### RequestObservation

- **Purpose**: Captures the stable dimensions needed to record request metrics and structured tracing for one HTTP exchange.
- **Fields**:
  - `route_label`
  - `method`
  - `status_class`
  - `duration`
  - `is_metrics_endpoint`
- **Rules**:
  - `route_label` must use a stable endpoint label rather than a raw URL with IDs.
  - `status_class` must distinguish success and error outcomes at minimum.
  - `is_metrics_endpoint` controls exclusion from normal request metrics.
  - Raw request IDs must never be used as metric labels.

## Derived Outputs

### Structured Request Event

- **Purpose**: A structured tracing event or span associated with a single request.
- **Composition**:
  - request correlation ID
  - method
  - route label
  - response status
  - elapsed duration
- **Rules**:
  - Events must be emitted for both successful and failed requests.
  - Sensitive headers, tokens, and internal error details must not be included.

### Prometheus Metrics Output

- **Purpose**: Text exposition of the service's current request metrics.
- **Composition**:
  - request count series
  - request latency series
  - request error series
- **Rules**:
  - Output is served by `/metrics`.
  - `/metrics` scrapes are excluded from the normal request metric series.
  - Metrics must remain available even when no recent business traffic exists.

## State Transitions

### Request Correlation Lifecycle

1. Receive inbound request.
2. Read inbound request ID header if present.
3. Select effective request ID by preserving the inbound value or generating a new one.
4. Attach the effective request ID to tracing context.
5. Return the effective request ID in the HTTP response.

### Request Metrics Lifecycle

1. Start request timing when the request enters the service.
2. Determine stable route and outcome labels when the response is produced.
3. Skip normal request metrics if the route is `/metrics`.
4. Record request count, latency, and error metrics for all other routes.
